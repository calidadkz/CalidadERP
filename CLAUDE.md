# CLAUDE.md — Инструкции для работы с CalidadERP

## О проекте

CalidadERP — торговая ERP-система для рынка Казахстана. React/TypeScript SPA с Supabase backend.

---

## Работа с claude-memory (Obsidian vault)

Папка `claude-memory/` — это Obsidian vault. Это **долгосрочная память проекта**.
Цель: наращивать граф знаний так, чтобы в начале каждой новой сессии можно было восстановить полный контекст.

### В начале каждой сессии — ОБЯЗАТЕЛЬНО:
1. Прочитать `claude-memory/Session-Log.md` — понять, что было сделано в прошлый раз и что осталось незавершённым
2. При необходимости прочитать связанные заметки (Architecture.md, Tech-Stack.md и тематические файлы по модулям)

### В конце каждой сессии — ОБЯЗАТЕЛЬНО:
1. Добавить запись в `claude-memory/Session-Log.md` по шаблону:
   ```
   ## YYYY-MM-DD — Название задачи
   **Что сделано:** ...
   **Файлы изменены:** ...
   **Открытые задачи / следующий шаг:** ...
   **Ссылки:** [[связанная заметка]]
   ```
2. Если в ходе сессии появились новые знания о модуле или архитектуре — обновить или создать соответствующую заметку в vault

### Как наращивать граф:
- Каждый значимый модуль или концепция должны иметь **свою заметку** в `claude-memory/`
- Использовать **Obsidian-ссылки** `[[Название заметки]]` для связывания заметок между собой
- Если работали над модулем (например, Bundles) — создать/обновить `claude-memory/Modules/Bundles.md`
- Если появилось решение нетривиальной задачи — зафиксировать в `claude-memory/Decisions/` как архитектурное решение (ADR)
- Если обнаружен баг или known issue — зафиксировать в `claude-memory/Issues/`

### Структура vault (целевая):
```
claude-memory/
├── Architecture.md          # Общая архитектура проекта [[Tech-Stack]]
├── Tech-Stack.md            # Стек технологий
├── Session-Log.md           # Хронологический лог сессий
├── Modules/                 # Заметки по каждому модулю
│   ├── Inventory.md
│   ├── Bundles.md
│   ├── Finance.md
│   └── ...
├── Decisions/               # Архитектурные решения (ADR)
│   └── 001-inventory-balance-types.md
└── Issues/                  # Известные баги / ограничения
```

### Принципы ведения заметок:
- Писать так, чтобы **Claude мог прочитать и сразу понять контекст** без объяснений пользователя
- Фиксировать **почему** было принято то или иное решение, не только что
- Ссылки `[[файл]]` создают граф — используй их активно
- Не дублировать то, что есть в коде — фиксировать то, что **не видно из кода** (намерения, компромиссы, контекст)

## Правила работы с кодом

### Язык
- UI-текст, комментарии в коде — **русский** (там, где уже используется русский)
- Технические идентификаторы (переменные, функции, типы) — **camelCase английский**
- DB-поля — **snake_case английский** (конвертация автоматическая через ApiService)

### Архитектурные принципы
- **Не добавлять** новые зависимости без явного запроса
- **Не рефакторить** код, который не относится к задаче
- **Не добавлять** docstrings/комментарии к коду, который не менялся
- Бизнес-логика живёт в `services/`, не в компонентах
- Состояние — в GlobalStore через специализированные хуки (`useInventoryState`, `useFinanceState` и т.д.)

### Ключевые паттерны
- API: все ключи автоматически конвертируются camelCase ↔ snake_case через `ApiService`
- Защищённые JSON-поля (не конвертируются): `matrix`, `machineConfig`, `priceOverrides`, `composition`, `configuration`, `settings`, `items`, `packingList`, `options`
- Optimistic locking: передавать `expectedUpdatedAt` в `ApiService.update()` для конфликт-детекции
- Валюты: KZT, USD, CNY, EUR, RUB — все денежные расчёты через `MoneyMath`
- Права доступа: проверять через `PermissionsService.canSee()` / `canWrite()`

### UI
- Иконки: только `lucide-react`
- Стили: Tailwind CSS v4 (utility-first, без кастомных CSS классов если можно избежать)
- Форматирование дат и чисел — с учётом локали KZ
