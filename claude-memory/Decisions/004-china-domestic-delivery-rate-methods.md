# ADR-004: Методы расчёта Доставки по Китаю
> Связанные заметки: [[Architecture]] | [[Session-Log]] | [[Modules/Batches-PreCalc]] | [[Decisions/001-rename-expense-categories-and-delivery]]
> Дата: 2026-04-08
> Статус: Принято

## Контекст

«Доставка по Китаю» (`logistics_china_domestic`) — это внутрикитайская логистика: перевозка товара от завода-производителя до Урумчи (или до точки погрузки). В отличие от «Доставки Урумчи–Алматы» (`shippingChinaUsdPerM3`), ставка на этот маршрут сильно варьируется:
- одни поставщики считают по объёму (м³)
- другие — по весу (тонна)
- для некоторых позиций удобнее ставить фиксированную цену за единицу

Ранее поле `deliveryChinaDomesticKzt` в предрасчётах отсутствовало (отложено 4+ сессии). Категория `logistics_china_domestic` в расходах партий уже существовала, но в расчётах предрасчёта не участвовала.

## Рассмотренные варианты

1. **Единый рейт за м³** (как у Урумчи–Алматы) — просто, не требует выбора метода. Минус: не подходит для поставщиков с весовым тарифом или фиксированной ценой; пришлось бы вводить «псевдообъём» вручную.

2. **Три метода с переключателем** (`volume` / `weight` / `fixed`) — чуть сложнее в реализации. Плюс: точно отражает реальные договорённости с перевозчиками, не требует обходных хаков.

3. **Ручной ввод на каждую позицию** — максимальная гибкость, но неудобно при десятках позиций в предрасчёте.

## Принятое решение

**Вариант 2 — три метода с переключателем в GeneralSettings.**

Метод задаётся на уровне предрасчёта (не на уровне позиции), так как в рамках одной партии перевозчик один и метод единый. Курс пересчёта используется общий с Урумчи–Алматы (`exchangeRateForShipping`), чтобы не плодить лишних полей.

Формулы:
- `volume`: `volumeM3 × ratePerM3Usd × exchangeRateForShipping`
- `weight`: `(weightKg / 1000) × ratePerTonUsd × exchangeRateForShipping`
- `fixed`: `fixedKztPerUnit` (фиксированная цена за единицу товара)

Включена в:
- `fullCostKzt` — полная себестоимость
- `preSaleCostKzt` — себестоимость до продажи
- `customsNdsKzt` — база таможенного НДС (с коэффициентом ×0.6, как у Урумчи)

## Последствия

**Плюсы:**
- Точный расчёт для разных схем работы с перевозчиками
- Не нужно вводить «фиктивный объём» для весового тарифа
- При создании партии `plannedLogisticsChinaDomesticKzt` теперь реальный, а не 0

**Минусы / компромиссы:**
- Метод задан на весь предрасчёт — если в одной партии разные товары везут разными схемами, придётся выбрать один метод или использовать `fixed` с ручным выравниванием
- Запоздалое отражение фактической оплаты (оплата за текущую партию попадает в следующую) — организационный процесс, не требует доп. кода; в Партиях категория `logistics_china_domestic` уже есть
